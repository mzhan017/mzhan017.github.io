# The Power of AI: Can We Only Let AI Thrive Even More? (2026-3-14)
I recently came across an article stating that Vibe Coding is contaminating open-source code
## Summary of following article
https://grith.ai/blog/vibe-coding-killing-open-source
The rise of AI is transforming the open-source ecosystem, yet data reveals its impacts are complex. Stack Overflow saw a 25% drop in user activity within six months of ChatGPT's launch, with weekly posts plummeting by half. AI-generated code is rife with issues, producing 1.7 times more problems per pull request (PR) than human-written code and being far more likely to be revised or reverted in the short term. Research also found that developers using AI tools actually took 19% longer to complete tasks—they expected a 24% speedup and subjectively perceived a 20% improvement after use.
Open-source projects have responded with a series of restrictions: Gentoo Linux prohibits AI-generated contributions entirely, NetBSD requires prior written approval for such submissions, and Mesa 3D mandates that contributors take full responsibility for AI-assisted code. Maintainers complain about the proliferation of low-quality code and a drastically increased review burden. The core issue is that while AI can serve as a valuable tool to assist development, it becomes problematic when used as a substitute for genuine understanding—yielding code that appears valid on the surface but is inherently low-quality, and breaking the "contribution-review" balance at the heart of the open-source community. If this trend continues, projects unable to adapt may cease maintenance, leaving AI tools without the open-source foundation they rely on. 
## My personal view
This summary is bound to spark a range of reflections, including the question posed in the article's title. So let’s get back to the core issue: how can humans ensure the correctness of AI-generated code? Especially given the massive volume of AI output—manual code review of such a scale would demand an enormous amount of work. What’s more, humans would need to possess knowledge equivalent to that of AI, as well as the soft skills such as the ability to connect disparate knowledge points, just to understand the code. This is virtually impossible for the average developer.
Of course, throwing the baby out with the bathwater is not the answer; a blanket ban on AI participation is unadvisable.
## So what will the end result be?
Perhaps we need more AI Agents—and crucially, a diverse array of them, isolated to develop distinct functional "personalities" and kept independent from one another, essentially enabling cross-validation. For instance:
Dedicated AI Agents for code review, tasked with auditing code quality;
Specialized testing AI Agents to verify the correctness of code generated by coding-focused Agents;
AI Agents for security testing to identify and mitigate security vulnerabilities;
End-user simulation AI Agents to conduct usability testing aligned with real user needs.
Only then can humans truly entrust AI with various tasks. And there’s no turning back—developers have already grown reliant on AI tools (echoing the question of Autonomous AI and its power). It’s like boarding a ship with no easy way off; there are simply no better alternatives. We need to foster the further prosperity of AI and accelerate its development to meet the demands of its massive output volume.
